# Codex CLI 更新チェック

- 最大で１日１回の自動更新処理は cmoc に組み込んでおきたい

# `<cmoc-branch>` をヒントとして渡す

- 「今の作業と関係のある変更は `<cmoc-branch>` に乗ってるからそれ見てね」を各コマンドでやりたい
- 主に「どこ見ていいか分からないから全チェック」を回避するのに使いたい
- 具体的にどういう注入のしかたするのかは gpt と相談

# `cmoc apply` の作業フェーズを細分化する

- 多分、差分調査を `eval-oracles` と同様に `oracles` ファイルごとの呼び出しにするのが良い
- oracles ベースとは別に実装ファイルベースのループも必要
    - `oracles` ベースの整合性チェックは既にやっているけど、実装ベースバグチェックはやっていない
    - 「ファイルごとに問題点を列挙」を `oracles` と実装とでやることで計算オーダーを N+M で実装起点のチェックを実行する
- 修正点リストアップ結果のリファクタも必要
    - 生の修正点リストはファイル単位の調査結果を元に好き勝手言ってるだけなので、修正点同士の整合性が保証されていない
    - 修正点リスト全体としての整合性を取る作業を実装作業とは独立した Codex CLI 呼び出しに括りだすことで、実装作業のコンテキストを節約（トークン消費を節約）する
- 修正点１つごとに修正作業の Codex CLI 呼び出しを行う
    - 別の作業は別のセッションとすることで、作業間でのコンテキストを持ち越さないようにする
    - それによりコンテキストの圧迫を緩和し、トークン消費の節約を狙う

# `cmoc apply` にも部分・全体モードを持ち込む

- 部分モードの場合、修正点リストアップのチェック対象を `<cmoc-branch>` 上で変化があったものに絞る
- `cmoc eval-oracle` と思想を揃える
- 「`<cmoc-branch>` 上で変化があったら」は頻出なので１箇所で定義してそれをあちこちで使いまわしたい

# 読むべきドキュメントの候補を機械的に列挙してプロンプトに注入する

- 今は、何を読むかは AI 任せになっているが、この判定処理にもトークンが消費される
- ワード検索などで機械的に列挙してファイルリストをプロンプトに注入することで、トークン消費の緩和を狙う
- 過去の Codex CLI 呼び出し履歴から、実際に行われているテクニックを抽出する

# モデル品質指定を仕様化

- 難易度に応じてモデルの重さを変える
- `codex_call.md` に判断例付きで

# cmoc の設定をファイルに出したい

- `<repo-root>/cmoconfig.json` に設定を書き出す
- `cmoc init` でデフォルト設定を書き出す

# テスト実行パス

- 汚しても良いところでトイプログラムで実行ということにする
- パスとかトイプログラムの仕様とかを決めないといけない

# Codex CLI 呼び出し時のサンドボックスモード指定

- サンドボックスモード指定を仕様化してなかったかも？
- Strctured Output を使えって書いただけだと --structued-output を使ってくれてないかも
    - https://zenn.dev/explaza/articles/251023-vibecoding-with-specification

# `eval-oracles` のレポートが読みづらい

- フォーマットがバラバラなのが原因
- `eval-oracles` のレポートも Structured Output にしたい
- 含めるべき項目は 過去の実行結果を元に AI に提案させる

# `eval-oracles` で読むべきファイルのリストをより正確にリストアップしたい

- `INDEX.md` とは別の、依存関係グラフが必要
- 依存関係情報を `INDEX.md` に書き込むのもアリかもしれない（see also として役に立つはず）
- 「意味的に関係があるか？」の判断のコアは Codex CLI になるが、機械的に処理可能な部分の責任は cmoc が持ち、 Codex CLI が担う責任範囲は最小化する

# バグ修正は正式なワークフローとして組み込んでも良いかも

- バグ修正は状態遷移的な操作なので、既存の oracles - apply ループには乗せにくい
- `oracles/bugs` とかに過去のやらかし事例を集積し、それらと同じ轍を踏んでいない事を apply の過程でチェック、とかだろうか

# `eval-oracles` に用語統一機能を組み込みたい

- 用語リストに基づいたファイル別チェックを駆使すれば、かなり安く行けるはず。
- 修正が漏れても良いタイプなので、 reasoning effort は低くても大丈夫なはず

# `cmoc status` コマンドが欲しい

- 現在のブランチが cmoc branch か
- base commit は何か
- 未コミット差分が oracle 内だけか
- 最後の eval-oracles / apply レポートはどれか
- apply が完了扱いか未完了扱いか
- merge 可能そうか

# `cmoc dector` コマンドが欲しい

- codex が呼べるか
- Structured Output が使えるか
- .cmoc が gitignore されているか
- oracles/INDEX.md があるか
- 必要な git 状態か
