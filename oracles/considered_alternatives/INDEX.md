# `memory_alternative.md`

## Summary

- AI-generated kaizen を次回の実行コンテキストへ自動的に持ち越さない方針を説明する文書です。
- 改善案の自動注入は、曖昧な準仕様レイヤーや誤診断の固定化を招くため採用しない、という判断を整理しています。
- cmoc では暗黙記憶よりも、`INDEX`・`oracles`・ログ・実行成果物のような明示的な情報源を優先する立場を示しています。

## Read this when

- AI が生成した改善案を次回以降の Codex CLI 呼び出しへ自動注入する設計を採用すべきか判断したいとき
- 仕様・ログ・成果物のような明示情報と、暗黙記憶レイヤーのどちらを優先すべきか検討したいとき
- `oracles` を正本仕様として維持する観点から、memory 的な仕組みの採否を整理したいとき

## Do not read this when

- `cmoc` の session / apply / eval-oracles など、この文書に直接関係しないサブコマンド仕様を確認したいとき
- `INDEX.md` の生成ルールや、`oracles` 全体のルーティング方針だけを確認したいとき
- memory 系の一般論ではなく、個別の実装ファイルやテストファイルの仕様だけを確認したいとき

# `working_plan_review.md`

## Summary

- `tgbt plan` や `/plan` のような計画レビュー方式を採らなかった理由を説明している。
- 人間が最終成果物にだけ関心を持つこと、AI へ丸投げしにくい現実、そして人間と AI を同列に扱う設計の不自然さを論じている。
- 結論として、人間が `oracles` を編集し、AI が実装を追従させるスタイルを採用した背景をまとめている。

## Read this when

- 作業計画レビューを採用しなかった理由を確認したいとき
- 人間が `oracles` を編集し、AI が実装を追従する方針の背景を知りたいとき
- 人間と AI を同じ土俵で共同作業させる設計の問題点を把握したいとき

## Do not read this when

- 人間と AI の役割分担をどう設計するかを議論したいとき
